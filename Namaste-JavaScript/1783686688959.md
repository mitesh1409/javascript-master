**Callback Hell**

Callback Hell = Callback Pyramid of Doom = Multiple levels of nested callbacks

Lets take an example of an Ecommerce application,  
where we need to do the following operations:  
* create an order
* take delivery address
* display order preview
* proceed for payment
* order successful notification

```javascript
// Following is more like a psuedocode not actual code to showcase the callback hell problem.

const cartItems = ['T-Shirt', 'Shirt', 'Shoes', 'Jeans', 'Watch'];

function createOrder(cartItems, takeDeliveryAddress) {
    // Business logic to create an order goes here.

    if (error) {
        // Error handling

        return;
    }

    // order created successfully

    takeDeliveryAddress(orderPreview) {
        // Business logic to take delivery address goes here.

        if (error) {
            // Error handling

            return;
        }

        // Delivery address set successfully

        orderPreview(orderPayment) {
            // Business logic to preview order goes here.

            if (error) {
                // Error handling

                return;
            }

            // Order preview done successfully

            orderPayment(orderSuccessNotification) {
                // Business logic for order payment goes here.

                if (error) {
                    // Error handling

                    return;
                }

                // Order payment successful

                orderSuccessNotification();
            }
        }
    }
}
```

"Inversion of Control" is another problem while we use callbacks.

Kyle Simpson explains this in his book series "You Don't Know JS Yet" - [You Don't Know JS Yet (book series) - 2nd Edition](https://github.com/getify/you-dont-know-js).

"Inversion of Control" is like you lose the control of your code while using callbacks.  
This is because we have no control over the callback, how and when it will be called by the higher order function is not in our control.
